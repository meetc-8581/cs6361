  1. Introduction

  1. Purpose

This document intends to capture the team's initial efforts to gain a better knowledge of the Preliminary Project Plan, which served as the foundation for the software development for the Theia Project, as well as the results of those efforts. It provides information on the problems that were experienced, the solutions that were examined to solve them, the decisions that were ultimately made, and the justifications for each choice.

  1. Scope

This document describes a smartphone app that enables a blind people to navigate indoors. The primary function of the application is to provide directions/voice commands to the user while navigating indoor spaces. As blind person who needs to move from one location to another location within a building faces challenges. The safety of the user would be one of the primary concerns of this app. For instance, If the user meets with an accident, the application will assist in an emergency.         

As smartphones have become so commonplace in recent years, the program specifically targets that market. The users will find this application more helpful than the present traditional assistance because they might not need to buy new gadgets.

2. Issues with Preliminary Definition Given (ambiguities, incompleteness, inconsistency, conflicts, ...)

        

2.1 Issues with II.1 The Domain, Stakeholders, Functional and Non-Functional Objectives 

2.1.1 Issue-1:​​The process for setting the configuration of the app by the caretaker.

Issue Description: According to the preliminary definition, some secondary stakeholder might involve setting up basic configuration of the app. But, what if the caretaker is not available with the blind person vehicle initial configuration of the app and account. What is the level of interaction and intervention we expect the caretaker to be involved at ?

Options:

  1. If a caretaker is available, then they can configure the app.  
  2. By using the voiceover (using accessibility) features of the smartphone, blind people can      configure the app by themselves.

 

Decision and Rationale: The app shall accommodate both the options as we discussed blind person should be able to configure the app by themselves and using the caretakers app. If the caretaker is available, then configuration time will be saved.

2.1.2 Issue-2: How does the application provide “safe and comfortable navigation”?

Issue Description: To provide safe and comfortable navigation, we have to add some way to detect and alert the blind person about obstacles like a person, maintenance cart, tables, chairs, etc. Hence, we have to support object detection and for that the user needs to hold the device such that the rear camera is pointing towards the path the user is going.

Options:

  1. If the user wants object detection then they should hold their smartphone according to the guidelines.
  2. If the user opts out of the object detection and holds their phone in a different way then only voice commands will work.

        

Decision and Rationale: By making use of cameras of the smartphone, the system can provide object detection and the application can assist users about objects which are on their way by voice commands.

2.1.3 Issue-3: Issues with voice communication?

Issue Description: What if the people are using the app and for some reason the only point of communication doesn’t work?

Options:

  1. We can use the haptic feedback provided by the devices and designate some haptic patterns to the different signs we need to give to the user while navigating.
  2. We can shut down the application as the user might not be able to interact with it.

Decision and Rationale : If voice communication fails, haptic commands will be helpful to the user.

2.1.4 Issue-4: Ambiguous definition of input method.

Issue Description: The definition of the non visual based input and interaction medium with the app is not clearly defined.

Options:

  1. By making use of a voice over os feature, the app will allow users to interact with the application.
  2. The user should be able interact with the application with a combination of touch input and haptic feedback from the application.

Decision and Rationale: The application will utilize both the methods and provide redundancy. In case of either one of them fails, the user will still be able to communicate with the application.

        

2.2 Issues with II.2 Software System Requirements: Functional Requirements 

2.2.1 Issue-1: Incomplete definition of how the user enters the destination location.

        

Issue description : Here it is not clearly defined how the app accepts the destination location. It also says the app may utilize the user’s routine schedule but what if the user wants to go to some different location rather than a routine location. Should User be able to enter the location by searching ?

Options:

  1. The app will suggest some locations based on user habits. The user will select one of them.
  2. The user will be able to enter the location manually through either voice input or via accessibility typing.
  3. System might use the user's daily routine schedule but the user should be informed before finalizing the destination location.

Decision and Rationale: A combination of all the options. The user shall use the recommendation or they can opt to enter some other location manually.

2.2.2 Issue-2: Ambiguity on how the routes will be calculated and curated to the user. How these routes will be selected by the user.

        

Issue description : It is uncertain how many options will be given to the user for possible routes to the destination. How much control the user/app will have to select the best route.

        

Options:

I. Multiple routes can be prompted to the user.

II. Shortest path will be selected by the app.

Decision and Rationale:

SInce it is an indoor navigation, the shortest path option would be the best for the user.

2.2.3 Issue-3: Failure in dialing emergency call in case of emergency

Issue description : Emergency calls and messages will be triggered possibly after detecting a fall or when the system cannot figure out the current location but if at that time sim card provided does not have service range or mobile pack of user is exhausted

Options :

I. Ring a loud alarm on the app to alert the surroundings and bring it to attention.

II. Users can dial 911 without carrier availability as well.

Decision and Rationale:

The app shall ring an alarm to notify the surroundings and all phones have the feature to contact emergency services without the carrier availability using the accessibility feature in the user's device.

2.2.4 Issue-4: Figuring out what would be the next action, based on the user’s schedule or habit.

        

Issue description : Here it is ambiguous what is the next action, is it the actions the app provides to the user while navigating or predicting and recommending the next destination location the user might want to go according to their schedule.         

        

Options:

  1. Store the locations that the user visits at a particular time. Use this data to recommend the next location the user might want to visit at the same time next week.
  1. Using the stored location, provide shortcuts to set the destination. Users can use these shortcut inputs to set their destination with minimal interaction.
  1. Depending on the destination entered by the user at a particular time, compute the optimal path and direct the user to the destination depending on their current location.
  1. Track user location through the GPS system built into the mobile phone. Depending on this location, direct the user on the next action that they need to take in order to reach their destination.
  2. Depending on the distance to the point that they need to take the next action (Eg. turn left, turn right, go straight), provide advance prompts on how far they need to walk before taking that action.

Decision and Rationale: The mobile app would use previous locations, chosen destinations, GPS position, and best path calculation to deliver personalized suggestions and navigation directions with prompts.

2.3 Issues with II.3 Software System Non-Functional Requirements 

        

2.3.1. Issue-1: Difficulty in the usability of the app

Issue description: It is important that special attention is paid on user experience because most potential users may have difficulty with their eye sight of various degrees and it should incorporate features that make it easy to use.

Options :

I. Enabling voice recognition because that is the ideal solution for a visually impaired candidate to be able to use a mobile application.

Ii. Users should refer user guide/manual for new users

Decision and rationale:

Both the solutions will be implemented to provide the user a good user experience.

        

2.3.2. Issue-2: User preferences - Customizability

Issue description: A user must be able to set the configurations of the application according to their convenience.

Options:

I.  Allowing the user to customize elements like languages for both user input and the system output.

Ii. Having rigid configurations made assuming one set of choices based on the majority of the users.

 

Decision and rationale

Going with option 1 and enabling the user to pick from multiple languages

2.3.3. Issue-3: User’s phone is lost or misplaced - security

Issue description: For instance, if the user has misplaced the phone, to avoid the misuse of the user information if the device is found by someone else, and disable anybody besides the user and the caretaker to access the emergency contact.

Options:

I. The application may sense and authenticate the user based on the biometrics at regular intervals

II. The application may use voice recognition to identify the bona fide user.

III. The application may request for the user credentials every time it is to be used.

IV. The application may authenticate  the user based on the biometrics when accessing or editing some personal or emergency contact information.

Decision and rationale:

A combination of option 2 and 4 will be implemented. The second option is the most feasible because having to authenticate the user constantly via biometric will need the user to hold the phone in their hands at all times, and asking to type in the user credentials conflicts with the ease of usability of the project. Voice recognition is the best solution because it will allow you to have your hands free.

2.3.4. Issue-4: User meeting with an accident - safety

Issue description: In case the app detects an accident, then how to know if it was a genuine one or a false alarm and how to handle both cases?

Options:

I. send out an alarm to the emergency contact selected by the user and/or the ambulance.

II. Ring a loud alarm on the app to alert the surroundings and bring it to attention.

III. If the app detects that the user fell down, it will give a prompt to the user to call the emergency contact.

Decision and rationale :

The best option will be to provide the prompt and default to calling the emergency contacts and giving the option to the user to cancel it within a timeout if an accident is detected by the device.

3. WRS

        

3.1 W

        

3.1.1 Problem 

People with impaired or lack of vision face a lot of difficulties when navigating indoors. This problem is even more prominent when you are trying to walk into crowded corridors. Managing to avoid obstacles and staying on the right path to the destination can be a very challenging, risk prone and overwhelming task. Additionally, when a blind person has lost their way it can be very difficult to get back on track relaying on the braille markings on the entrances of doors.

        

3.1.2 Goal

Our goal is to create a smartphone app which provides a very easy and intuitive way to help blind and visually impaired people. With the help of touch, button press and auditory input and haptic and auditory output form the mobile device. This application intends to minimize the difficulties faced by blind people navigate indoors.

                

3.1.3 Improved understanding of II.1 The Domain, Stakeholders, Functional and

Non-Functional Objectives

  1. (DRS1)The Theia application will be used by people (henceforth be referred to as blind people), regardless of the level of their deficiency, with vision disorders or impairments.

  1. (DRS2)The Theia application shall aid blind people navigate in at least one of the floors of the ECSS, ECSN and ECSW buildings.

  1. (DRS3)The system shall provide the blind person with non visual based input methods to input and interact with the application.

  1. (DRS4)The system shall provide only auditory and tactual sensation to the user when providing directions to the user.

  1. (DRS5)The system shall provide a special input method or interface to the user for making emergency calls, these emergency services may include but is not limited to 911, assistive services, hospitals or special emergency contacts.

  1. (DSR6)The system shall allow the summoning of an assistive person.
  1. (DSR6.a) The user shall be able request summoning of an assistive person for carrying out a task.        
  2. (DSR6.b) The blind person shall be able to request an action of the assistive person using the device.

3.2 RS

                

3.2.1 Functional RS – Improved understanding of II.2 Software System Requirements: FRs

  1. (FRS1)The system shall provide the following sensory aid to the blind person when entering their intended destination.
  1. (DRS1.a) An auditory confirmation of the location they intend to reach.
  2. (DRS1.b) A haptic response on destination confirmation.

  1. (FRS2) The system shall provide a unique auditory and tactual sensation when the user has arrived at their intended destination.

  1. (FRS3) The system shall include an accessible button on the home screen of the application to initiate emergency calls. This may include calling 911, to an emergency contact, contacting assistive services or to send a message to a nearby department such as hospital.

        

        

3.2.2 Non-functional RS - Improved understanding of II.2 Software System Requirements: NFRs

  1. (NFRS1) The system shall allow the blind person to easily and intuitively enter the intended location they want to reach.

  1. (NFRS2) The system shall offer ease of use by having a less steep learning curve.

  1. (NFRS3) The system shall provide safety to the user by detection and prompting of obstacles in the user’s path.

  1. (NFRS4) The system shall offer redundancy in the method of communication.
  1. In case of the voice I/O system failure, there needs to be other mechanisms in place which helps the user to communicate with the app.

  1. (NFRS5) The system shall provide usage of multiple languages in the app.

  1. (NFRS6) The system shall make the user account available to users regardless of device change.
